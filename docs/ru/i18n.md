# Добавить язык

Новый язык — это **контент + конфиг + один файл i18n. Правки шаблонов не нужны.**
Ни партиала-зеркала, ни собственного переключателя, ни своего формата даты.

Готовый пример — само демо: английский на `/`, русский на `/ru/`, одно дерево
контента, одна тема, ноль переопределений. Всё, о чём ниже, видно в
[`exampleSite/`](../../exampleSite/); начинать удобно с
[`exampleSite/hugo.toml`](../../exampleSite/hugo.toml) — он прокомментирован
ровно под это.

## Шаги {#steps}

### 1. Объявить язык {#declare}

По блоку `[languages.<код>]` на язык. Язык по умолчанию живёт без префикса,
остальные — под `/<код>/…`.

```toml
defaultContentLanguage = "en"
defaultContentLanguageInSubdir = false   # en на /, а не на /en/

[languages.en]
  languageCode = "en"
  languageName = "English"
  weight = 1
  [languages.en.params]
    heroLine  = "a theme for learning guides: series, widgets in the text, search"
    heroSat   = "· this demo is written in it, about it"
    footerTag = "the theme's demo site"
  [[languages.en.menus.main]]
    name = "How-to"
    pageRef = "/howto"
    weight = 10
  # … остальное меню и menus.footer

[languages.ja]
  languageCode = "ja"
  languageName = "日本語"
  weight = 2
  # свои params и menus
```

`languageName` — то, чем переключатель называет язык (уходит в `title`/`aria-label`,
а видимая подпись — код заглавными). Языковые `params` и `menus` перекрывают
общесайтовые, поэтому повторять нужно только то, что действительно различается:
в демо это `heroLine`, `heroSat`, `footerTag` и два меню. Всё остальное
(`defaultTheme`, `featured`, `rubricSections`, `ogImages`, …) остаётся в `[params]`
и общее для всех языков.

### 2. Переводить контент суффиксом файла {#content}

Hugo связывает страницу с переводами по суффиксу — руками ничего сшивать не надо:

```
content/howto/get-started/index.md      # en (по умолчанию)
content/howto/get-started/index.ru.md   # ru
```

Так же для `_index.md` рубрики, `_index.md` серии и главной. Остальные ресурсы
бандла (`cover.png`, `widgets.js`, …) — общие, см. [§7](#widgets).

В демо слаг и теги намеренно одинаковы во всех языках: гайд и его перевод делят
путь (`/howto/get-started/` ↔ `/ru/howto/get-started/`), и переключатель прыгает
между ними, не теряя места. Непереведённая страница тоже не тупик — переключатель
уводит на главную того языка. (Для большого числа языков у Hugo есть штатная
альтернатива: свой `contentDir` на язык и параллельные деревья.)

### 3. Дать строки интерфейса {#strings}

Тема везёт [`i18n/en.toml`](../../i18n/en.toml) и
[`i18n/ru.toml`](../../i18n/ru.toml) — **по 106 ключей, наборы совпадают**. Для
любого другого языка скопируйте один из них в `i18n/<код>.toml` своего сайта и
переведите. Hugo мерджит ваш файл поверх темы, так что неполный файл — это
нормально: чего нет, подставится из темы.

Сборка с `--printI18nWarnings` покажет, каких ключей ещё не хватает. В CI демо
это идёт вместе с `--panicOnWarning`, поэтому забытый ключ роняет сборку.

Множественные числа — по CLDR: перечислите те формы, которые в языке есть, а
нужную Hugo выберет сам.

```toml
# i18n/ru.toml
[guides_count]
one   = "{{ .Count }} гайд"
few   = "{{ .Count }} гайда"
many  = "{{ .Count }} гайдов"
other = "{{ .Count }} гайда"
```

Две группы внутри этих 106 ключей заслуживают отдельных разделов: 26 ключей с
префиксом `js_` ([§4](#js)) и двенадцать `month_*` ([§5](#dates)).

### 4. Строки, живущие в JS {#js}

Модалка поиска, поповер темы, кнопка фокуса, миникарты панелей и фильтр тегов
собирают разметку в браузере — их строки шаблон отрендерить не может. Берутся они
всё равно из вашего файла i18n.

[`layouts/_partials/js-bridge.html`](../../layouts/_partials/js-bridge.html) —
последнее, что делает `scripts.html`, — читает эти ключи и отдаёт их как
`window.THEME_I18N`, а
[`assets/js/modules/i18n.js`](../../assets/js/modules/i18n.js) мерджит объект
поверх своих английских дефолтов. Префикс `js_` помечает ровно те ключи, что
уезжают в браузер (их 26, плюс переиспользованный `mins_abbr` — в JS он
`minutes`).

**Чтобы их перевести, ничего особенного делать не нужно** — это обычные ключи
`i18n/<код>.toml`:

```toml
[js_search_placeholder]
other = "найти гайд, серию, тему…"
[js_search_empty]
other = "Ничего не нашлось. Попробуйте «gc», «slice», «планировщик»…"
```

**Чтобы переопределить одну строку, не трогая каталог**, выставите
`window.THEME_I18N` из хук-партиала. Мост мерджит поверх того, что оставил хук, —
`Object.assign(каталог, window.THEME_I18N || {})`, — поэтому ваши ключи побеждают
поштучно, а остальной каталог остаётся на месте:

```html
<!-- layouts/_partials/hooks/head-extra.html (или hooks/foot-extra.html) -->
<script>window.THEME_I18N={searchPlaceholder:"найти гайд…",pinPanel:"закрепить"};</script>
```

Имена здесь — **JS-имена, а не ключи i18n**: `searchPlaceholder`, а не
`js_search_placeholder`. Имя, которого нет в каталоге, молча ничего не делает,
поэтому берите их из словаря в `js-bridge.html` (это и есть полный список) или из
`DEFAULTS` в `assets/js/modules/i18n.js` — набор тот же. Основной бандл
отложенный (`defer`), так что инлайн-скрипт из любого хука отработает раньше него.

Множественные числа в JS — тоже CLDR, с обеих сторон. Фильтр тегов считает гайды
в браузере, и фразу Hugo отрендерить не может; вместо этого Hugo рендерит по слову
на пробное число (1, 2, 5) в `guideForms`, а `plural(n, forms)` спрашивает у
`Intl.PluralRules` — те же данные CLDR, по `<html lang>` — какое из пробных чисел
попадает в ту же категорию, что и настоящее. Языку с другими правилами нужны
только собственные формы `js_guides_word`:

```toml
[js_guides_word]
one   = "гайд"
few   = "гайда"
many  = "гайдов"
other = "гайда"
```

### 5. Даты {#dates}

[`layouts/_partials/date.html`](../../layouts/_partials/date.html) собирает
короткую дату ленты (`27 июн`, `27 Jun`, плюс год, если он не текущий) из ключей
`month_1` … `month_12`. Это намеренно не локальный форматтер: русскому нужна
родительная короткая форма, которую лента проговаривает вслух (`30 мая`, а не
`май`), а ни один форматтер её не даёт.

Так что даты нового языка — это ещё двенадцать строк в его файле i18n. Партиала
даты, который надо переопределять, больше нет.

### 6. Данные по языкам {#data}

`data/` в Hugo не знает про языки, поэтому роадмап разрешается в двух формах:

| Сайт | Раскладка |
| --- | --- |
| один язык | `data/roadmap.toml` — плоский файл, как и было |
| многоязычный | `data/roadmap/<язык>.toml` — по файлу на язык |

В форме с папкой Hugo отдаёт шаблонам карту с ключами по именам файлов, то есть по
кодам языков. И [`layouts/_shortcodes/roadmap.html`](../../layouts/_shortcodes/roadmap.html),
и [`layouts/_partials/home/wip.html`](../../layouts/_partials/home/wip.html) берут
`site.Data.roadmap` и, если в нём есть ключ, равный текущему `site.Language.Lang`,
используют эту вложенную карту. Работают обе формы, и флага в конфиге не нужно ни
одной. В демо лежат
[`exampleSite/data/roadmap/en.toml`](../../exampleSite/data/roadmap/en.toml) и
[`ru.toml`](../../exampleSite/data/roadmap/ru.toml).

### 7. Виджеты общие для переводов {#widgets}

У бандла гайда ровно **один `widgets.js`**. Никакого `widgets.ru.js` в Hugo нет:
ресурсы принадлежат бандлу, а не переводу, поэтому `index.md` и `index.ru.md` из
одной папки обслуживает один и тот же скрипт.

Двуязычный виджет поэтому ветвится по языку документа:

```js
// один ресурс бандла обслуживает оба перевода — строки выбираем по <html lang>
var RU = document.documentElement.lang.indexOf("ru") === 0;
var L = RU
  ? { add: "+ раздел", none: "Гайд без разделов…" }
  : { add: "+ section", none: "A guide with no sections…" };
```

Ровно так устроен каждый виджет демо — короче всего
[`exampleSite/content/howto/first-guide/widgets.js`](../../exampleSite/content/howto/first-guide/widgets.js).
В `<html lang>` лежит `languageCode` страницы, так что проверка работает на любой.
Ветвить нужно только строки внутри скрипта: разметка виджета и его статический
фолбэк живут в теле гайда, а это такой же контент, как всё остальное, — свой на
каждый перевод.

## Что делать не нужно {#free}

Всё перечисленное начинает работать в тот момент, когда в конфиге появился второй
язык. Ни параметра, ни шаблона это не требует:

- **Переключатель языков.**
  [`layouts/_partials/lang-switch.html`](../../layouts/_partials/lang-switch.html),
  вызываемый из `header.html`, появляется сам, как только в `site.Languages`
  больше одной записи, — и не рендерит ничего, если язык один. Каждая ссылка
  ведёт на перевод *этой* страницы, а если перевода нет — на главную того языка.
- **`<html lang>` и `og:locale`** следуют за языком страницы.
- **Альтернативы `hreflang` и `x-default`** выдаются на каждой странице
  многоязычного сайта из
  [`head/meta.html`](../../layouts/_partials/head/meta.html); на одноязычном не
  выдаётся ничего.
- **Поиск.** Pagefind сам читает `<html lang>` и строит индекс по языкам.
  Настройки языка поиска нет (как нет и выключателя поиска).
- **RSS.** По ленте на язык: `/index.xml`, `/ru/index.xml`.

Ни русской строки, ни зашитого `lang` в шаблонах искать не придётся: вся
оставшаяся в `layouts/` кириллица — в комментариях.

## Про CJK {#cjk}

Японский и китайский устроены точно так же; сверх общего им нужны три ресурса,
которые тема принести за вас не может.

- **Шрифты.** В Inter и JetBrains Mono нет ни каны, ни иероглифов. Подключите
  сабсет Noto (или аналога) для текста — `@font-face` в `custom.css` или в хуке
  `head-extra.html` — и TTF для обложек.
  [`og-image.html`](../../layouts/_partials/og-image.html) ищет шрифт по имени
  сперва в папке стиля `assets/og/<стиль>/`, затем в `assets/og/fonts/`: положите
  TTF в любую из них и назовите его в `layout.toml` стиля. Языку, которому нужна
  своя типографика обложек, можно завести отдельную папку стиля и выбирать её
  постранично через `og_style` во front matter.
- **Pagefind.** Язык индекса он берёт из `<html lang>` без всякой настройки, но
  сегментатор CJK есть только в **extended**-сборке (`pagefind_extended`).
  Стандартная, которую ставит CI демо, покрывает латиницу и стемминг русского.
  Вся правка — заменить бинарник; в теме не меняется ничего.
- **i18n.** Файл `i18n/<код>.toml`, как в [§3](#strings).
