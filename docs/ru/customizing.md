# Кастомизация без форка

Всё, что ниже, делается **на стороне твоего сайта**: параметр, файл данных,
CSS-файл, одноимённый партиал. Тема при этом не правится ни разу, поэтому её
обновление остаётся обычным `git pull` или бампом версии модуля. Порядок —
примерно от самого дешёвого к самому тяжёлому.

## 1. Параметры

Поведение и тексты — это параметры (полный список: [params.md](params.md)).
Задаются в `[params]` твоего `hugo.toml`. Прежде чем браться за что-то тяжелее,
посмотри, нет ли готового параметра.

## 2. custom.css — переопределить любой стиль

Если у сайта есть `assets/css/custom.css`, тема подключает его **последним** в
CSS-бандл — твои правила выигрывают каскад без форка и без `!important`:

```css
/* assets/css/custom.css */
:root { --content: 820px; }          /* колонка текста пошире */
.article h2 { letter-spacing: -0.01em; }
```

### Стабильные CSS-переменные (публичный API токенов)

Эти имена стабильны — опирайся на них, а не на внутренние селекторы.

Структура (`assets/css/00-tokens.css`):

| Переменная | По умолчанию | Что это |
|---|---|---|
| `--shell` | `1104px` | Максимальная ширина широких страниц (главная, рубрики) |
| `--content` | `760px` | Ширина колонки текста |
| `--rail` | `236px` | Ширина боковых рельсов |
| `--font-sans` / `--font-mono` | Inter / JetBrains Mono | Шрифтовые стеки |
| `--radius-sm` / `--radius` | `8px` / `12px` | Скругления |
| `--dur` / `--ease` | `.14s` / `cubic-bezier(.2,0,0,1)` | Длительность и кривая переходов |

Цвета приходят из **палитры** (см. ниже) и потому меняются вместе с ней:
`--bg-deep`, `--bg-base`, `--bg-surface`, `--bg-raised`, `--bg-hover`,
`--border`, `--text-primary`, `--text-secondary`, `--text-muted`, `--text-ghost`,
`--text-strong`, `--gtok-str` и акценты `--accent` (+ `-dim`, `-glow`),
`--accent-green` (+ `-dim`, `-glow`), `--accent-copper` (+ `-dim`),
`--accent-blue` (+ `-dim`), `--accent-gold` (+ `-dim`).

Главный `--accent` — особый: сайт перекрашивает его сразу во всех палитрах одним
параметром, см. [Перекрасить акцент](#recolour).

## 3. Палитра — это данные {#palettes-are-data}

Палитра — один файл, `data/themes/<id>.toml`: служебные поля плюс цветовые
переменные (ключ = имя CSS-переменной без `--`):

```toml
# data/themes/mine.toml
name = "Моя"         # имя в пикере
weight = 25          # позиция в списке пикера
"bg-base" = "#101418"
"accent"  = "#7aa2f7"
# … остальные ключи — как в data/themes/amber.toml
```

Положи файл в **свой** `data/themes/` — палитра сама появится и в CSS, и в пикере
в шапке; тему править не нужно. Hugo мержит данные темы и сайта (при совпадении
имён побеждает сайт), поэтому так же можно **переопределить** палитру из комплекта
(тот же файл, другие значения) или **выключить** её (`disabled = true`).

`name` — подпись в пикере: обычная строка или таблица переводов по языкам,
`name = { en = "Mine", ru = "Моя" }`. Пикер берёт запись текущего языка, а если
её нет — `en`. Живёт в файле палитры, не в `i18n/`: палитра остаётся одним
самодостаточным файлом.

Служебные ключи — `name`, `weight`, `light`, `disabled` — в CSS-переменные не
превращаются никогда. Всё остальное в файле — превращается.

### Перекрасить акцент {#recolour}

**Одна строка — все палитры сразу.** Акцент — это брендовая ось, идущая *поперёк*
палитр, поэтому один параметр перекрашивает его везде:

```toml
# hugo.toml
[params]
accent = "#7aa2f7"
```

Теперь каждая палитра из комплекта — и любая твоя — берёт этот цвет вместо
родного: ссылки, кнопки, активные состояния, акцентная точка на свотче в пикере,
крона логотипа и она же в фавиконе.

Родной акцент — ледниковая бирюза, и это **не один хекс**: тёмные палитры несут
`#55c0b7`, а `Light ☀` — `#0e7a71`. Это не разнобой, а вынужденность. Акцент
служит ещё и *фоном* кнопки (`.btn-cta` пишет по нему цветом `--bg-deep`), значит
на тёмной палитре он обязан быть достаточно светлым, чтобы держать почти чёрный
текст, а на светлой — достаточно тёмным, чтобы держать почти белый. Эти два
требования не пересекаются, поэтому один тон живёт в двух светлотах. Задав
`accent` сам, ты схлопываешь обе в одно значение — проверь его на светлой палитре,
прежде чем выкатывать.
`--accent-dim` и `--accent-glow` выводятся из него (`rgba(…, .18)` и
`rgba(…, .28)`). Если выведенная прозрачность не устраивает — задай явно:

```toml
accent     = "#7aa2f7"
accentGlow = "rgba(122, 162, 247, .45)"   # ореол посильнее
```

`accent` обязан быть шестизначным `#rrggbb`; любое другое значение роняет сборку.

Логотип (`layouts/_partials/logo.html`) — инлайновый SVG, его крона читает
`var(--accent)` и потому вживую следует за палитрой, которую выбрал читатель.
Фавикон так не умеет: это data-URI, а data-URI не видит CSS-переменных — цвет его
кроны **запекается на сборке** из `params.accent`, а без него берётся акцент
палитры `params.defaultTheme`. При переключении палитры фавикон не меняется.

**Куда этот рычаг не дотягивается:**

- **Графика обложек OG** — цвет, живущий в `base.png`, запечён. А вот *текстовые*
  блоки могут следовать оси: блок раскладки с `color = "accent"` запекается уже
  разрешённым брендовым цветом (так устроен кикер штатного `dots`; кикер `taiga`
  нарочно остаётся литералом — его бирюза принадлежит картинке). Для полного
  ребренда всё равно вези свою папку [стиля обложек](#og-cover-styles), а не жди,
  что параметр перерисует картинку.
- **Вторичные акценты** (`--accent-green`, `--accent-copper`, `--accent-blue`,
  `--accent-gold`) и цвет строк в коде `--gtok-str` — переменные палитры, а не
  часть единой оси акцента. Их меняют в файлах палитр (см. выше).

**Разный акцент в разных палитрах.** Если хочется, чтобы у каждой палитры был
*свой* акцент, а не один общий, — не задавай `accent`, правь файлы палитр. Акцент
там — ключ `accent` (плюс `-dim` / `-glow`) в `data/themes/<id>.toml`; вези свою
палитру или переопредели готовую (см. выше). У всех семи встроенных акцент сейчас
одинаковый, так что перекрашивать придётся каждый файл — или только те, которыми
пользуешься. Фавикон при этом всё равно берёт один цвет — акцент палитры по
умолчанию.

## 4. Данные роадмапа {#roadmap-data}

Шорткод `roadmap` и полоска «в работе» на главной читают один файл данных, и
принадлежит он **сайту**. Форм у него две:

```
data/roadmap.toml         # один язык
                          # — или —
data/roadmap/en.toml      # по файлу на язык
data/roadmap/ru.toml
```

`data/` в Hugo не знает про языки, поэтому шаблоны разбираются сами: берут
`site.Data.roadmap` и, если там есть ключ, равный коду текущего языка, работают с
этой вложенной картой, иначе — с картой целиком. Плоский файл поэтому продолжает
работать как работал, а папка даёт каждому языку свой экземпляр. Язык, для
которого файла в папке нет, просто не получает блоков роадмапа — молча, не ошибкой.

В любой из форм шаблоны читают три таблицы: `[[items]]` (`title`, `status`,
`progress`, `note`, `ready`), `[[queue]]` (`title`) и `[[rules]]` (`title`,
`note`). В полоску на главной попадают те `item`, у которых нет `ready`. Рабочая
пара файлов — в [`exampleSite/data/roadmap/`](../../exampleSite/data/roadmap/);
шорткод, который всё это рисует, — в [authoring.md](authoring.md).

## 5. Hook-партиалы — вставить своё в head/foot {#hooks}

Тема вызывает два пустых партиала, которые сайт наполняет сам
(`layouts/_partials/hooks/`):

- `head-extra.html` — аналитика, свои meta-теги, шрифты. Рендерится последним в
  `<head>`.
- `foot-extra.html` — скрипты в конец body.

Создай тот же путь в своём `layouts/_partials/hooks/` — и содержимое подхватится.
Включать ничего не нужно.

```html
<!-- layouts/_partials/hooks/head-extra.html -->
<script defer src="https://analytics.example/script.js"></script>
```

## 6. Переопределить любой партиал

Lookup order в Hugo всегда предпочитает файлы сайта файлам темы, поэтому
одноимённый путь в твоём `layouts/` заменяет партиал темы. Партиалы мелкие и
именованные ровно для этого — `logo.html`, `footer.html`, `post.html` (карточка
ленты), `home/logos/<section>.html` (глиф рубрики на карточках главной; без него
тема берёт `home/logos/default.html`) и так далее. Скопируй версию темы,
поправь — готово.

Один партиал несущий: `scripts.html` заканчивается строкой
`{{ partial "js-bridge.html" . }}`, которая отдаёт в страницу `window.THEME_I18N`
(все строки интерфейса, живущие в JS) и `window.TAIGA_PAGEFIND` (адрес бандла
поиска, разрешённый через `baseURL`). Переопределяешь `scripts.html` — сохрани эту
строку. Запасные варианты есть у обоих значений, и оба хуже настоящих: интерфейс
застынет на английских дефолтах темы, а поиск откатится на корневой
`/pagefind/pagefind.js`, который на сайте в подпапке отдаёт 404 — молча, под видом
подсказки «индекс ещё не собран».

## 7. i18n — переписать строки интерфейса {#i18n}

Любая строка интерфейса — ключ i18n. Положи в свой сайт частичный `i18n/ru.toml`
(или `en.toml`) только с теми ключами, которые меняешь; Hugo смержит его поверх
файла темы поключево — переписывать каталог целиком не нужно.

```toml
# i18n/ru.toml
[read_guide]
other = "читать гайд →"
```

Добавить целый язык (а не переписать строки одного) — задача другая, она в
[i18n.md](i18n.md).

### Строки, живущие в JS {#js-strings}

Модалка поиска, поповер «Тема/Вид», кнопка фокуса, минимапы рельсов и фильтр на
`/tags/` собирают свой текст в браузере. Но это **всё те же ключи i18n**: ключи
`js_*` из каталога (плюс `mins_abbr`), а `layouts/_partials/js-bridge.html`
отдаёт их в страницу как `window.THEME_I18N`.

Поэтому обычный способ поменять JS-строку — обычный: переопредели ключ `js_*` в
своём `i18n/<lang>.toml`. Никакого JavaScript:

```toml
# i18n/ru.toml
[js_search_placeholder]
other = "найти гайд…"
```

**Аварийный люк на рантайме.** Мост подмешивает свой каталог *под* уже
существующий объект:

```js
window.THEME_I18N = Object.assign({ …каталог темы… }, window.THEME_I18N || {});
```

значит, твои ключи побеждают — если твой скрипт отработал **раньше**. Так и будет,
если это **инлайновый** `<script>` в любом из хуков (`head-extra.html` или
`foot-extra.html`): оба рендерятся до моста, а собственный бандл темы висит на
`defer` — инлайн всегда успевает первым. Скрипт с `defer`/`async` уже опоздал:
бандл читает каталог в момент загрузки.

```html
<!-- layouts/_partials/hooks/foot-extra.html -->
<script>window.THEME_I18N = { themeHead: "Цвета", pinPanel: "не прятать" };</script>
```

Имена ключей в JS **не совпадают** с именами ключей i18n. Полный контракт:

| Ключ в JS | Ключ i18n | Где виден |
|---|---|---|
| `themeHead` | `js_theme_head` | Поповер: заголовок «Тема» |
| `viewHead` | `js_view_head` | Поповер: заголовок «Вид» |
| `railLeft` | `js_rail_left` | Поповер: переключатель левого рельса |
| `railRight` | `js_rail_right` | Поповер: переключатель правого рельса |
| `lists` | `js_lists` | Поповер: группа стилей списков |
| `on` / `off` | `js_on` / `js_off` | Поповер: состояния переключателя |
| `listRows` | `js_list_rows` | Поповер: стиль списка «строки» |
| `listLoud` | `js_list_loud` | Поповер: стиль списка «громко» |
| `listShelf` | `js_list_shelf` | Поповер: стиль списка «полка» |
| `popoverAria` | `js_popover_aria` | `aria-label` кнопки поповера |
| `hint` | `js_hint` | Подсказка по клавишам в поповере (несёт разметку `<kbd>`) |
| `pinPanel` | `js_pin_panel` | Минимапа рельса: кнопка закрепления |
| `keyOr` | `js_key_or` | Минимапа рельса: «или» между двумя сочетаниями |
| `focusEnter` | `js_focus_enter` | Плашка выхода в режиме фокуса |
| `focusTitle` | `js_focus_title` | `title` кнопки фокуса |
| `focusAria` | `js_focus_aria` | `aria-label` кнопки фокуса |
| `searchAria` | `js_search_aria` | `aria-label` модалки поиска |
| `searchPlaceholder` | `js_search_placeholder` | Плейсхолдер поля поиска |
| `searchEmpty` | `js_search_empty` | Строка «ничего не найдено» |
| `searchUnbuilt` | `js_search_unbuilt` | Показывается, когда индекс Pagefind не собран |
| `scrollTop` | `js_scrolltop` | Кнопка «наверх» |
| `minutes` | `mins_abbr` | Единица времени чтения в выдаче поиска |
| `tagFeedHead` | `js_tag_feed_head` | `/tags/`: заголовок над отфильтрованной лентой |
| `tagReset` | `js_tag_reset` | `/tags/`: сброс фильтра |
| `tagEmptyTail` | `js_tag_empty_tail` | `/tags/`: хвост строки «такого тега нет» |
| `guideForms` | `js_guides_word` | `/tags/`: слово «гайд» в нужном числе |

`guideForms` стоит особняком: гайды считает браузер, поэтому фразу Hugo отрендерить
не может. Он рендерит *слово* на пробных числах 1, 2 и 5 и отдаёт объект
`{ "1": …, "2": …, "5": … }`; в рантайме `Intl.PluralRules` решает, какое из
пробных чисел попадает в ту же форму множественного, что и настоящее. Переопредели
`js_guides_word` в `i18n/<lang>.toml` (обычные формы `one` / `few` / `many` /
`other`) — остальное произойдёт само. Если всё-таки переопределяешь через
`window.THEME_I18N`, сохрани форму объекта.

Английские строки, зашитые в `assets/js/modules/i18n.js`, — аварийный запас на
случай, когда мост не отработал (сайт заменил `scripts.html`), а не источник правды.

## 8. Стили обложек OG {#og-cover-styles}

Стиль обложки — это **папка** в `assets/og/`; имя папки и есть его id. Свой стиль
везётся без форка: положи `assets/og/<mystyle>/` в свой сайт.

```
assets/og/mystyle/
├── base.png       # подложка 1200×630 (вся графика)
├── layout.toml    # раскладка текстовых блоков (см. assets/og/dots/layout.toml)
└── home.png       # необязательно: готовый постер для не-гайдовых страниц
```

Выбирается на весь сайт через `params.ogImages.style = "mystyle"` или на странице
через `og_style:`. Готовый стиль так же переопределяется одноимённой папкой.
(`images.Text` не умеет letter-spacing и выравнивает по правому краю только
моноширинный текст — оба ограничения честно описаны в `layout.toml` штатного стиля
`dots`.)

`color` у блока в `layout.toml` — либо литерал `#rrggbb`, либо слово `accent`:
брендовый акцент, разрешаемый на сборке ровно как у фавикона — сначала
`params.accent`, иначе акцент палитры `params.defaultTheme`.

## 9. Свой раздел, которого в теме нет

Тема не несёт ничего тематического — и всё же сайт может собрать целый авторский
раздел из одних штатных механизмов. «Песочница» на демо — разобранный пример:

- **страница** — `content/playground/index.md`, обычный Markdown, обычный бандл, с
  большим интерактивом через шорткод `widget` и `widgets/w-reflex/widget.js` в
  том же бандле (механизм виджетов работает для любого бандла, не только для
  гайдов);
- **бейдж в меню** — это `[[menus.main]]` с `params.badge = "β"`: генерик-фича
  навигации, а не «фича песочницы»;
- **стили только для этого раздела**, если они вообще нужны, живут в том же бандле
  как `widgets/w-reflex/widget.css` (или `widgets/_shared/shared.css`, если
  виджетов больше одного) — тема подключает их ровно на этой странице и не
  несёт таких стилей у себя;
- в поиске и в sitemap это просто ещё одна страница.

Так собирается любой свой раздел: тема остаётся чистой, а сайт — насколько угодно
своим.
