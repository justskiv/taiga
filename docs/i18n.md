# Adding a language

Adding a language is **content + config + one i18n file. No template edits.** No
partial to mirror, no switcher to write, no date format to fork.

The demo is the worked example: English at `/`, Russian at `/ru/`, one content
tree, one theme, zero overrides. Everything below is visible in
[`exampleSite/`](../exampleSite/) — start from
[`exampleSite/hugo.toml`](../exampleSite/hugo.toml), which is commented for
exactly this.

## Steps {#steps}

### 1. Declare the language {#declare}

A `[languages.<code>]` block per language. The default language has no path
prefix; the others live under `/<code>/…`.

```toml
defaultContentLanguage = "en"
defaultContentLanguageInSubdir = false   # en at /, not /en/

[languages.en]
  languageCode = "en"
  languageName = "English"
  weight = 1
  [languages.en.params]
    heroLine  = "a theme for learning guides: series, widgets in the text, search"
    heroSat   = "· this demo is written in it, about it"
    footerTag = "the theme's demo site"
  [[languages.en.menus.main]]
    name = "How-to"
    pageRef = "/howto"
    weight = 10
  # … the rest of the menu, and menus.footer

[languages.ja]
  languageCode = "ja"
  languageName = "日本語"
  weight = 2
  # its own params and menus
```

`languageName` is what the switcher announces (it rides in `title`/`aria-label`;
the visible label is the uppercased code). Per-language `params` and `menus`
override the site-wide ones, so restate only what actually differs — in the demo
that is `heroLine`, `heroSat`, `footerTag` and the two menus. Everything else
(`defaultTheme`, `featured`, `rubricSections`, `ogImages`, …) stays in `[params]`
and is shared.

### 2. Translate content by file suffix {#content}

Hugo pairs a page with its translations by the file suffix. Nothing links them by
hand:

```
content/howto/handbook/get-started/index.md      # en (default)
content/howto/handbook/get-started/index.ru.md   # ru
```

Same for rubric `_index.md`, series `_index.md` and the home `_index.md`.
The bundle's other resources (`cover.png`, the `widgets/` folder, …) are shared
— see [§7](#widgets).

In the demo the slug and the tags are deliberately identical across languages, so
a guide and its translation share a URL path (`/howto/handbook/get-started/` ↔
`/ru/howto/handbook/get-started/`) and the switcher hops between them in place. An
untranslated page is not a dead end either: the switcher falls back to that
language's home. (For many languages, a per-language `contentDir` with parallel
trees is Hugo's documented alternative.)

### 3. Provide UI strings {#strings}

The theme ships [`i18n/en.toml`](../i18n/en.toml) and
[`i18n/ru.toml`](../i18n/ru.toml) — **100 keys each, identical key sets**. For any
other language, copy one of them into your site's `i18n/<code>.toml` and
translate. Hugo merges yours over the theme's, so a partial file is fine:
whatever you leave out falls back to the theme's string.

Build with `--printI18nWarnings` to see which keys you still owe. The demo's CI
runs it with `--panicOnWarning`, so a missing key fails the build.

Plurals are CLDR: list the forms your language actually has and Hugo picks the
right one.

```toml
# i18n/ru.toml
[guides_count]
one   = "{{ .Count }} гайд"
few   = "{{ .Count }} гайда"
many  = "{{ .Count }} гайдов"
other = "{{ .Count }} гайда"
```

Two groups inside those 100 keys deserve their own sections: the 16 keys prefixed
`js_` ([§4](#js)) and the twelve `month_*` keys ([§5](#dates)).

### 4. Strings that live in JS {#js}

The search modal, the theme popover, the focus button, the rail minimaps and the
tags filter build their markup in the browser, so their strings cannot be rendered
by a template. They still come from your i18n file.

[`layouts/_partials/js-bridge.html`](../layouts/_partials/js-bridge.html) — the
last thing in `scripts.html` — reads those keys and emits them as
`window.THEME_I18N`; [`assets/js/modules/i18n.js`](../assets/js/modules/i18n.js)
merges the object over its own English defaults. The `js_` prefix marks exactly
which keys leave the server (16 of them, plus `mins_abbr`, reused as `minutes`).

**To translate them, do nothing special** — they are ordinary keys in
`i18n/<code>.toml`:

```toml
[js_search_placeholder]
other = "search a guide, a series, a topic…"
[js_search_empty]
other = "Nothing found. Try «gc», «slice», «scheduler»…"
```

**To override one string without touching the catalogue**, set
`window.THEME_I18N` from a hook partial. The bridge merges over whatever a hook
left there — `Object.assign(catalogue, window.THEME_I18N || {})` — so your keys
win one by one, and the rest of the catalogue survives:

```html
<!-- layouts/_partials/hooks/head-extra.html (or hooks/foot-extra.html) -->
<script>window.THEME_I18N={searchPlaceholder:"find a guide…",pinPanel:"pin"};</script>
```

The names here are the **JS names, not the i18n keys**: `searchPlaceholder`, not
`js_search_placeholder`. A name that is not in the catalogue is a silent no-op, so
take them from the dict in `js-bridge.html` — it is the whole list — or from
`DEFAULTS` in `assets/js/modules/i18n.js`, which is the same set. The main bundle
is deferred, so an inline script from either hook always runs before it.

JS plurals are CLDR on both sides. The tags filter counts guides in the browser,
so Hugo cannot render the phrase; instead Hugo renders one word per probe count
(1, 2, 5) into `guideForms`, and `plural(n, forms)` asks `Intl.PluralRules` —
the same CLDR data, keyed off `<html lang>` — which probe falls in the same
category as the real count. A language with different plural rules therefore needs
nothing but its own `js_guides_word` forms:

```toml
[js_guides_word]
one   = "гайд"
few   = "гайда"
many  = "гайдов"
other = "гайда"
```

### 5. Dates {#dates}

[`layouts/_partials/date.html`](../layouts/_partials/date.html) renders the short
feed date (`27 Jun`, `27 июн`, plus the year when it isn't the current one) from
the i18n keys `month_1` … `month_12`. It is not a locale formatter on purpose:
Russian wants the genitive short form the feed reads out loud (`30 мая`, not
`май`), and no formatter hands that over.

So a language's dates are twelve more strings in its i18n file. There is no date
partial to override.

### 6. Per-language data {#data}

Hugo's `data/` is not language-aware, so the roadmap resolves in two shapes:

| Site | Layout |
| --- | --- |
| one language | `data/roadmap.toml` — flat, unchanged |
| multilingual | `data/roadmap/<lang>.toml` — one file per language |

In the folder form Hugo hands the templates a map keyed by file name, i.e. by
language code. [`layouts/_shortcodes/roadmap.html`](../layouts/_shortcodes/roadmap.html)
and [`layouts/_partials/home/wip.html`](../layouts/_partials/home/wip.html) both
take `site.Data.roadmap` and, if it has a key equal to the current
`site.Language.Lang`, use that sub-map — so both shapes work and neither needs a
config flag. The demo ships
[`exampleSite/data/roadmap/en.toml`](../exampleSite/data/roadmap/en.toml) and
[`ru.toml`](../exampleSite/data/roadmap/ru.toml).

### 7. Widgets are shared across translations {#widgets}

A leaf bundle has exactly **one `widgets/` folder**. Hugo has no `widgets.ru.js`
and no per-language `widgets/` variant: bundle resources belong to the bundle,
not to a translation, so `index.md` and `index.ru.md` in the same folder are
served by the same `widgets/<id>/widget.js`.

A bilingual widget therefore branches on the document language:

```js
// one bundle resource serves both translations — pick the strings by <html lang>
var RU = document.documentElement.lang.indexOf("ru") === 0;
var L = RU
  ? { add: "+ раздел", none: "Гайд без разделов…" }
  : { add: "+ section", none: "A guide with no sections…" };
```

Every widget in the demo does exactly this —
[`exampleSite/content/howto/handbook/first-guide/widgets/w-mins/widget.js`](../exampleSite/content/howto/handbook/first-guide/widgets/w-mins/widget.js)
is the shortest one to copy. `<html lang>` carries the page's `languageCode`, so
the check works on any page. Only the strings inside the script need branching:
the widget's markup and its static fallback live in the guide body, and that is
per-translation like any other content.

## What you don't have to do {#free}

Everything in this list already works the moment a second language exists in the
config. None of it is a param, and none of it needs a template:

- **The language switcher.**
  [`layouts/_partials/lang-switch.html`](../layouts/_partials/lang-switch.html),
  rendered from `header.html`, appears by itself as soon as `site.Languages` has
  more than one entry — and renders nothing at all otherwise. Each link prefers
  *this page's* translation and falls back to that language's home.
- **`<html lang>` and `og:locale`** follow the page's language.
- **`hreflang` alternates and `x-default`** are emitted on every page of a
  multilingual site by [`head/meta.html`](../layouts/_partials/head/meta.html);
  a single-language site emits none.
- **Search.** Pagefind reads `<html lang>` and builds a per-language index on its
  own. There is no search language setting (and no search on/off switch either).
- **RSS.** One feed per language: `/index.xml`, `/ru/index.xml`.

There is no Russian string and no hardcoded `lang` in the templates to chase — the
only Cyrillic left in `layouts/` is inside comments.

## CJK note {#cjk}

Japanese and Chinese work exactly the same way; what they additionally need are
three assets the theme cannot ship for you.

- **Fonts.** Inter and JetBrains Mono carry no kana, kanji or hanzi. Ship a Noto
  (or similar) subset for the body text — an `@font-face` in `custom.css` or in
  the `head-extra.html` hook — and a TTF for the OG covers.
  [`og-image.html`](../layouts/_partials/og-image.html) resolves a font by name
  from the style folder `assets/og/<style>/` first, then from
  `assets/og/fonts/`: drop the TTF in either and name it in the style's
  `layout.toml`. A language that wants its own cover typography can get its own
  style folder, selected per page with `og_style` in the front matter.
- **Pagefind.** It picks the index language off `<html lang>` with no config, but
  the CJK segmenter only ships in Pagefind's **extended** build (`pagefind_extended`).
  The standard build the demo's CI installs covers Latin and Russian stemming.
  Swapping the binary is the whole change; nothing in the theme moves.
- **i18n.** An `i18n/<code>.toml`, as in [§3](#strings).
